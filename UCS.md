## UC-01 · 회원가입

| 항목 | 내용 |
|------|------|
| **ID** | UC-01 |
| **이름** | 회원가입 |
| **액터** | 일반 사용자 |
| **설명** | 사용자가 이메일 주소와 비밀번호로 계정을 생성한다. |
| **사전 조건** | 사용자가 서비스에 가입되어 있지 않다. |
| **사후 조건** | 계정이 생성되고, 사용자는 로그인할 수 있는 상태가 된다. |

**기본 흐름**

1. 사용자가 회원가입 폼에 이메일·비밀번호·비밀번호 확인을 입력한다.
2. 시스템이 이메일 형식과 비밀번호 규칙(8자 이상, 영문+숫자 포함)을 검증한다.
3. 시스템이 이메일 중복 여부를 확인한다.
4. 시스템이 비밀번호를 해시(bcrypt)로 암호화하여 DB에 저장한다.
5. 시스템이 "회원가입이 완료되었습니다." 메시지를 반환한다.

**대안 흐름**

- **2a.** 이메일 형식이 잘못된 경우 → `400 이메일 형식이 올바르지 않습니다.` 반환 후 종료.
- **2b.** 비밀번호 규칙 미충족 → `400 비밀번호는 8자 이상이며 영문과 숫자를 포함해야 합니다.` 반환 후 종료.
- **3a.** 이미 가입된 이메일 → `409 이미 사용 중인 이메일입니다.` 반환 후 종료.

## UC-02 · 로그인

| 항목 | 내용 |
|------|------|
| **ID** | UC-02 |
| **이름** | 로그인 |
| **액터** | 일반 사용자, 관리자 |
| **설명** | 등록된 계정으로 인증하고 JWT 토큰을 발급받는다. |
| **사전 조건** | 사용자가 회원가입을 완료한 상태이다. |
| **사후 조건** | Access Token과 Refresh Token이 발급되고, 사용자는 인증이 필요한 API를 호출할 수 있다. |

**기본 흐름**

1. 사용자가 이메일·비밀번호를 입력하여 `POST /auth/login`을 호출한다.
2. 시스템이 DB에서 이메일로 사용자를 조회한다.
3. 시스템이 입력 비밀번호를 해시와 비교한다.
4. 시스템이 Access Token(15분)과 Refresh Token(7일)을 생성하여 반환한다.
5. 클라이언트가 이후 요청 헤더에 `Authorization: Bearer <token>`을 포함한다.

**대안 흐름**

- **2a.** 이메일이 존재하지 않는 경우 → `401 이메일 또는 비밀번호가 올바르지 않습니다.` 반환.
- **3a.** 비밀번호 불일치 → `401 이메일 또는 비밀번호가 올바르지 않습니다.` 반환.
- **4a.** 5회 연속 실패 시 → `429 로그인 시도 횟수를 초과했습니다. 10분 후 재시도하세요.`

## UC-04 · 단일 이메일 스팸 분류

| 항목 | 내용 |
|------|------|
| **ID** | UC-04 |
| **이름** | 단일 이메일 스팸 분류 |
| **액터** | 일반 사용자 |
| **설명** | 이메일 제목과 본문을 입력하면 spam / ham 여부와 신뢰도를 반환한다. |
| **사전 조건** | 사용자가 로그인 상태이며, 분류에 사용할 모델(ml 또는 bert)이 로드된 상태이다. |
| **사후 조건** | 분류 결과가 반환되고, 이력 DB에 저장된다. |

**기본 흐름**

1. 사용자가 `POST /predict`에 `subject`, `body`, `model`("ml" | "bert")을 전송한다.
2. 시스템이 전처리 모듈(Preprocessor)로 텍스트를 정제한다.
   (HTML 제거 → URL/이메일/전화번호 토큰화 → 특수문자 정리 → 공백 정규화)
3. 시스템이 선택된 모델로 예측을 수행한다.
4. 스팸 확률이 임계값(0.7) 이상이면 `spam`, 미만이면 `ham`으로 판정한다.
5. 시스템이 `label`, `confidence`, `spam_probability`, `ham_probability`, `processing_time_ms`를 반환한다.
6. 시스템이 요청·결과·사용자 ID를 이력 DB에 저장한다.

**대안 흐름**

- **1a.** 미인증 요청 → `401 인증이 필요합니다.`
- **2a.** subject 또는 body가 빈 문자열 → `422 제목과 본문을 입력해주세요.`
- **3a.** 요청 모델이 로드되지 않은 상태 → `503 해당 모델이 로드되지 않았습니다. 먼저 학습을 실행하세요.`

| 항목 | 내용 |

| **이름** | 대량 이메일 일괄 분류 |
| **액터** | 일반 사용자 |
| **설명** | 최대 100건의 이메일을 한 번의 요청으로 일괄 분류한다. |
| **사전 조건** | 사용자가 로그인 상태이며, 사용할 모델이 로드된 상태이다. |
| **사후 조건** | 각 이메일에 대한 분류 결과 목록과 전체 처리 시간이 반환된다. |

**기본 흐름**

1. 사용자가 `POST /predict/batch`에 이메일 목록(최대 100건)을 전송한다.
2. 시스템이 동일 모델 요청끼리 묶어 배치 추론을 수행한다.
3. 각 이메일에 대해 UC-04의 3~5 단계를 수행한다.
4. 전체 결과 목록과 `total_processing_time_ms`를 반환한다.

**대안 흐름**

- **1a.** 이메일 목록이 100건 초과 → `422 최대 100건까지 처리 가능합니다.`
- **1b.** 이메일 목록이 비어 있음 → `422 분류할 이메일이 없습니다.`

## UC-06 · Gmail 계정 연동 및 스팸 분류

| 항목 | 내용 |
|------|------|
| **ID** | UC-06 |
| **이름** | Gmail 계정 연동 및 스팸 분류 |
| **액터** | 일반 사용자, Gmail 서버 |
| **설명** | 사용자가 자신의 Gmail 계정을 OAuth 2.0으로 연동하여 실제 수신 메일을 가져오고 스팸 분류를 실행한다. |
| **사전 조건** | 사용자가 로그인 상태이며, 분류에 사용할 모델이 로드된 상태이다. Google Cloud Console에서 Gmail API가 활성화되어 있고 OAuth 2.0 클라이언트가 등록된 상태이다. |
| **사후 조건** | 가져온 메일 각각에 대한 스팸 분류 결과가 반환된다. 사용자의 OAuth 토큰이 서버에 암호화 저장된다. |

**기본 흐름**

**[연동 단계]**

1. 사용자가 `GET /gmail/connect`를 호출한다.
2. 시스템이 Google OAuth 2.0 인증 URL을 생성하여 반환한다.
   - 요청 스코프: `https://www.googleapis.com/auth/gmail.readonly`
3. 사용자가 브라우저에서 Google 로그인 및 권한 동의를 완료한다.
4. Google 서버가 인증 코드(Authorization Code)를 콜백 URL(`GET /gmail/callback`)로 전달한다.
5. 시스템이 인증 코드를 Access Token·Refresh Token으로 교환한다.
6. 시스템이 Gmail OAuth 토큰을 사용자 DB에 암호화하여 저장한다.
7. 시스템이 "Gmail 연동이 완료되었습니다." 메시지를 반환한다.

**[메일 가져오기 및 분류 단계]**

8. 사용자가 `POST /gmail/classify`에 조회 옵션을 전송한다.

   ```json
   {
     "max_results": 50,
     "label_filter": "INBOX",
     "model": "bert"
   }
   ```

9. 시스템이 저장된 OAuth 토큰으로 Gmail API를 호출하여 메일 목록을 가져온다.
10. 시스템이 각 메일의 제목(subject)과 본문(body)을 파싱한다.
    - base64 디코딩 처리
    - multipart 메일의 경우 `text/plain` 파트 우선 추출
    - HTML 태그 제거
11. 시스템이 UC-05(대량 일괄 분류)와 동일한 방식으로 스팸 분류를 수행한다.
12. 시스템이 아래 항목을 포함한 결과 목록을 반환한다.

    | 필드 | 설명 |
    |------|------|
    | gmail_id | Gmail 메시지 ID |
    | subject | 이메일 제목 |
    | sender | 발신자 주소 |
    | date | 수신 일시 |
    | label | spam / ham |
    | confidence | 예측 신뢰도 |
    | spam_probability | 스팸 확률 |

**[연동 해제 단계]**

13. 사용자가 `DELETE /gmail/disconnect`를 호출하면 시스템이 저장된 OAuth 토큰을 삭제하고 Google 토큰 폐기 API를 호출한다.

**대안 흐름**

- **1a.** 미인증 요청 → `401 인증이 필요합니다.`
- **3a.** 사용자가 Google 권한 동의를 거부 → `403 Gmail 접근 권한이 거부되었습니다.`
- **5a.** 인증 코드 만료 (10분 초과) → `400 인증 코드가 만료되었습니다. 다시 시도해주세요.`
- **9a.** OAuth 토큰 만료 시 → 시스템이 Refresh Token으로 자동 재발급 후 재시도.
- **9b.** Refresh Token도 만료된 경우 → `401 Gmail 재연동이 필요합니다.` 반환 후 연동 해제 처리.
- **9c.** Gmail API 호출 실패(네트워크 오류) → `502 Gmail 서버와 통신에 실패했습니다.`
- **10a.** 메일 본문 파싱 실패 시 → 해당 메일의 body를 빈 문자열로 처리하고 분류 계속 진행.
- **11a.** 모델이 로드되지 않은 상태 → `503 해당 모델이 로드되지 않았습니다. 먼저 학습을 실행하세요.`

## UC-09 · ML 모델 학습

| 항목 | 내용 |
|------|------|
| **ID** | UC-09 |
| **이름** | ML 모델 학습 (TF-IDF + Logistic Regression) |
| **액터** | 관리자 |
| **설명** | 데이터셋으로 경량 ML 모델을 학습하고 artifacts에 저장한다. |
| **사전 조건** | 관리자 권한으로 로그인한 상태이며, `data/korean_spam_dataset.csv`가 존재한다. |
| **사후 조건** | `artifacts/ml_model.joblib`이 저장되고, 성능 지표가 기록된다. |

**기본 흐름**

1. 관리자가 `POST /admin/train/ml`에 학습 파라미터를 전송한다.

   ```json
   {
     "max_features": 50000,
     "ngram_range": [1, 2],
     "C": 1.0,
     "spam_threshold": 0.7
   }
   ```

2. 시스템이 `data/korean_spam_dataset.csv`를 로드하고 전처리를 수행한다.
3. 시스템이 데이터를 train(80%) / val(10%) / test(10%)로 분리한다.
4. 시스템이 `MLSpamClassifier`를 학습시킨다 (`class_weight='balanced'` 적용).
5. 시스템이 validation 세트로 중간 결과를 출력한다.
6. 시스템이 test 세트로 최종 Precision·Recall·F1을 계산한다.
7. 시스템이 모델을 `artifacts/ml_model.joblib`으로 저장한다.
8. 시스템이 성능 지표를 `artifacts/ml_metrics.json`으로 저장한다.
9. 시스템이 학습 완료 메시지와 성능 요약을 반환한다.

**대안 흐름**

- **1a.** 일반 사용자 권한으로 접근 → `403 관리자 권한이 필요합니다.`
- **2a.** CSV 파일 없음 → `404 데이터셋 파일을 찾을 수 없습니다.`
- **4a.** 학습 중 오류 발생 → `500 학습 중 오류가 발생했습니다: {오류 메시지}`
- **4b.** 이미 학습 중인 작업 존재 → `409 이미 학습이 진행 중입니다.`

## UC-10 · DL(BERT) 모델 학습

| 항목 | 내용 |
|------|------|
| **ID** | UC-10 |
| **이름** | DL 모델 학습 (KR-ELECTRA BERT 파인튜닝) |
| **액터** | 관리자 |
| **설명** | 사전학습 KR-ELECTRA 모델을 스팸 분류 태스크로 파인튜닝하고 저장한다. |
| **사전 조건** | 관리자 권한으로 로그인한 상태이며, `data/korean_spam_dataset.csv`가 존재한다. CUDA GPU 사용 시 드라이버가 설치된 상태를 권장한다. |
| **사후 조건** | `artifacts/bert_model/`에 모델 파일이 저장되고, 성능 지표가 기록된다. |

**기본 흐름**

1. 관리자가 `POST /admin/train/bert`에 학습 파라미터를 전송한다.

   ```json
   {
     "base_model": "snunlp/KR-ELECTRA-discriminator",
     "max_length": 256,
     "batch_size": 32,
     "epochs": 5,
     "learning_rate": 2e-5,
     "warmup_ratio": 0.1,
     "weight_decay": 0.01,
     "spam_threshold": 0.7
   }
   ```

2. 시스템이 `data/korean_spam_dataset.csv`를 로드하고 전처리를 수행한다.
3. 시스템이 데이터를 train(80%) / val(10%) / test(10%)로 분리한다.
4. 시스템이 HuggingFace Hub에서 베이스 모델과 토크나이저를 다운로드한다.
5. 시스템이 에폭마다 아래 과정을 반복한다.
   - mini-batch 학습 (AdamW 옵티마이저, Linear Warmup 스케줄러)
   - validation F1 계산 및 콘솔 출력
   - 이전 최고 F1보다 높으면 체크포인트 저장 (Early Stopping 적용)
6. 학습 완료 후 best 체크포인트를 `artifacts/bert_model/`에 저장한다.
7. 시스템이 test 세트로 최종 Precision·Recall·F1을 계산한다.
8. 시스템이 성능 지표를 `artifacts/bert_metrics.json`으로 저장한다.
9. 시스템이 학습 완료 메시지와 에폭별 학습 곡선 데이터를 반환한다.

**대안 흐름**

- **1a.** 일반 사용자 권한으로 접근 → `403 관리자 권한이 필요합니다.`
- **2a.** CSV 파일 없음 → `404 데이터셋 파일을 찾을 수 없습니다.`
- **4a.** 네트워크 오류로 모델 다운로드 실패 → `502 베이스 모델 다운로드에 실패했습니다.`
- **5a.** GPU 메모리 부족(OOM) → batch_size를 절반으로 줄여 자동 재시도 (최대 1회).
- **5b.** 학습 중 오류 발생 → `500 학습 중 오류가 발생했습니다: {오류 메시지}`
- **5c.** 이미 학습 중인 작업 존재 → `409 이미 학습이 진행 중입니다.`
